### Why This Two-Step Approach?
- Design Mode is 3-5x cheaper on credits than App Mode
- You get the hardest part (visual design) right first
- Converting preserves all your design work
- App Mode follow-ups for functionality are smaller, cheaper operations

### Credit-Saving Tips
- **Don't interrupt the agent mid-build** — let it finish before giving feedback
- **Batch your feedback** — say "fix these 3 things" not 3 separate messages
- **Be specific** — "make the hero text 48px and left-aligned" burns fewer credits than "make it look better"
- **Use Economy Mode** if available in your settings

### Page 2: RideOps (/ride-ops)

This page should feel like a PRODUCT PAGE, not a portfolio entry. Lead with the problem story.

**Hero:** Page title "RideOps" with subtitle "Campus Transportation Ops Platform". A "LIVE PRODUCT" badge. Two buttons: [Visit ride-ops.com →] and [Open App Demo →].

**Stats strip:** 29 analytics widgets · 21 notification types · 4 campuses · 650+ seeded rides · 6-state ride lifecycle

**Three role cards in a row:** Admin (full command center: ride board, analytics dashboard, scheduling, fleet management) | Driver (mobile interface: clock in, pickup lifecycle, grace timer) | Rider (book in three taps)

**Story section — use this exact copy:**

USC's DART program — Disabled Access to Road Transportation — moves students with disabilities across campus by golf cart. When I joined as a student supervisor, the entire operation ran on Gmail and Google Calendar. Cancellations were deleted events. No-shows were strikethroughs nobody tracked. Drivers didn't know their schedules until the last minute. Managers couldn't answer basic questions without counting by hand.

The problem wasn't the people. It was the tools.

I built RideOps to fix that. It's a white-labeled, multi-tenant SaaS platform purpose-built for university accessibility transportation programs. Three separate interfaces — dispatcher, driver, rider — share a single PostgreSQL backend and stay in sync in real time.

The dispatcher gets a full command center: a real-time ride board, a 29-widget drag-and-drop analytics dashboard, shift scheduling, fleet management, and Excel report exports. The driver gets a focused mobile interface: clock in, see your rides, run the pickup lifecycle, and let the grace timer handle no-shows automatically. The rider books in three taps.

Every university gets its own branded experience — colors, logos, program name — via tenant config, not forked codebases. USC DART, UCLA BruinAccess, Stanford ATS, and UCI AnteaterExpress each look native to their campus. The multi-tenant architecture means I add a new school by adding a config entry, not a new deployment.

I showed a prototype to the USC Transportation General Manager in December 2025. He gave me a feature roadmap and told me schools at his conference network need this. UCLA, Stanford, and UCI have standing demo offers.

**Technical section:** Built for university IT: FERPA-compliant data handling, bcrypt session auth, RBAC, parameterized SQL, SECURITY.md for IT evaluators, and a full DPA framework.

**Story section — use this exact copy:**

Most MENA economic coverage is either paywalled institutional research or surface-level news aggregation. Neither is particularly useful if you want a fast weekly read on macro signals, trade flows, and labor trends across nine countries simultaneously.

MENA Rising is my attempt to build what I actually wanted to read.

Every week, a Python pipeline wakes up via GitHub Actions, pulls raw data from the World Bank, FRED, Finnhub, News API, and GDELT, and sends it to Claude's API to generate a structured briefing JSON. Subscribers receive a formatted HTML email after each pipeline run via a Resend-powered endpoint on Vercel.

**Copy:**
The Fama-French three-factor model is one of the foundational frameworks in empirical asset pricing. This notebook implements it end-to-end: raw data ingestion, factor construction (market excess return, SMB, HML), OLS regression across a portfolio of equities, and interpretation of the results.

Built in Python using pandas, statsmodels, and matplotlib. Data pipeline runs from source to analysis without manual steps.

The memo also analyzes Section 7874 of the Internal Revenue Code — the main U.S. legislative mechanism against corporate inversions — and explains why it has not prevented Microsoft and others from maintaining these structures.

### Page 8: DSCI 351 (/dsci-351)

**Title:** "Dual-Database Inventory Management System"
**Course:** DSCI 351 — Data Systems, USC
**Format badge:** Technical Report · 12 pages
**Tags:** [MongoDB] [Redis] [Python] [Systems Design] [Technical Writing] [CRUD] [GUI]

**Copy:**
A trader buys products from suppliers and sells them to retailers. The entire lifecycle — purchase, inventory, order, fulfillment — needs to be tracked in one place. This system does that, using two databases chosen for different access patterns: Redis for the buy-side (fast key-value reads on products and supplier data) and MongoDB for the sell-side (flexible document storage for orders, inventory, and retailer records).

The report documents the full build: ER diagram with five entity relationships, five Python backend modules, three complex MongoDB aggregate pipeline queries, and a Tkinter GUI with CRUD operations across all entities.

My contributions were the top-down architecture design, the justification for the Redis/MongoDB split, and all of the NoSQL query work — including three multi-stage aggregate pipelines for cross-entity queries that the flat Redis key-value structure couldn't handle directly.
